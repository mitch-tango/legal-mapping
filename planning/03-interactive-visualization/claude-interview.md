# Interview Transcript: Interactive Visualization & Export

## Q1: Graph Library Choice
**Q:** The research strongly favors Cytoscape.js over D3.js (your prototype's library) — it has built-in layouts, interactions, and easier SVG export. Are you comfortable switching?
**A:** Switch to Cytoscape.js (Recommended)

## Q2: Default Landing View
**Q:** When you open the visualization with a deal loaded, which view should be the default?
**A:** "What do you recommend?" — Architect recommendation: **Graph view (dagre hierarchy)** as default, with sidebar summary of critical findings.

## Q3: PDF Export Priority
**Q:** For PDF export to outside counsel — graph diagram, tables/checklists, or both?
**A:** Both equally important.

## Q4: Data Loading Workflow
**Q:** How should deal data get into the visualization?
**A:** Claude Code embeds data into HTML. Claude Code regenerates the HTML file with deal-graph.json and deal-analysis.json baked in. User just opens the HTML.

## Q5: Typical Deal Size
**Q:** How large are your typical deals?
**A:** Range from 5 to ~30 documents. No need for heavy clustering, but should handle 30 nodes gracefully.

## Q6: Browser Editing Priority
**Q:** How important is direct browser editing (relationships, annotations, flagging) for v1?
**A:** Nice-to-have, defer if complex. Start with view-only + PDF export; add editing later if feasible. Claude Code handles edits conversationally for now.

## Q7: Data Refresh Strategy
**Q:** When Claude re-runs analysis, regenerate the entire HTML file or support JSON reload?
**A:** Regenerate HTML each time. Simple workflow: Claude Code produces new HTML, user opens fresh.

## Q8: Timeline/Sequence View Format
**Q:** What format for the closing execution sequence view?
**A:** Simple table sorted by sequence — document name, sequence #, dependencies, status.

## Q9: PDF Branding
**Q:** Should PDF include cover page with branding?
**A:** Don't worry about it for now. Focus on content quality; branding can be added later.

## Q10: Conflict/Risk View Format
**Q:** Should conflicts be a separate view or overlay on the graph?
**A:** Overlay on the graph — toggle "show issues" mode that highlights problem nodes/edges with badges, keeps graph visible.

## Q11: Prototype UX Feedback
**Q:** Anything about the prototype's UX that worked well or didn't?
**A:** The detail sidebar was useful — clicking a node/edge and seeing metadata in a side panel worked well.

---

## Key Decisions Summary

- **Library:** Cytoscape.js (replacing D3.js from prototype)
- **Architecture:** Single HTML file with embedded data, regenerated by Claude Code
- **Default view:** Dagre hierarchy graph with findings sidebar
- **Editing:** Deferred to later phase; v1 is view-only + export
- **Conflicts:** Overlay on graph (toggle mode), not separate view
- **Timeline:** Simple sorted table, not Gantt chart
- **PDF:** Graph + tables equally important, no branding for now
- **Deal size:** 5-30 documents, no heavy clustering needed
- **Data flow:** Claude Code embeds → user opens → describes edits to Claude Code → regenerate
